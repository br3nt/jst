JST

JavaScript Templates (JST) — reactive components in plain HTML, with JavaScript itself as the templating language. No build step.

It's just Web Components under the hood — every <jst-*> is a real custom element from the platform's own customElements.define. JST renders them; what you ship is standard custom elements.

How does it work?

JST is a thin layer over the Web Components platform. A <script type="jst"> tag is compiled into a class and registered with customElements.define() — so your <jst-*> tags are genuine custom elements, not a framework's virtual stand-ins. You write them in ordinary HTML with JavaScript as the templating language — no compiler, no bundler, no JSX, no virtual DOM.

Because they're real custom elements, they behave like the platform: they show up in DevTools as <jst-counter>, you set data with plain properties (el.count = 5), you listen with addEventListener, they work inside any framework or none, and they upgrade automatically when their definition arrives — even from the server. JST adds rendering and ergonomics; the component is the browser's.

The hole it fills

JST started from a question: what would a front-end SSR component look like? Could the server send down components, like HTMX — but with real client-side interactivity, like Alpine — without a build step, a virtual DOM, or signals?

HTMXServer returns HTML; hypermedia-driven.No client-side interactivity without a round-trip; introduces its own attribute language.
AlpineSprinkle client interactivity into HTML.No server-streamed components; ad-hoc scopes and stores.

JSTComponents in plain HTML that the server can stream down (a fetched fragment can define and use a component — it auto-registers), with full client interactivity, no build, and no virtual DOM or signals — it rides the platform: custom elements, properties, events, DOM morphing.

We tested that claim: 70 canonical examples from HTMX, Alpine, Vue and React, rebuilt in JST. 35 were exact matches, 35 needed a documented workaround, none were impossible. The gaps are catalogued honestly.

Hypertext as your API — not JSON

A typical web app is a JSON API plus a separate client that turns that JSON back into HTML — every screen defined twice and kept in sync by hand. JST takes the older idea (HATEOAS): a response is the UI — HTML that carries its own next actions.

The part HTMX can't do: a JST response is HTML containing real <script type="jst"> components, so it brings its own front-end interactivity — the browser registers and runs the JS on arrival. You get HTMX's "the server just returns HTML" simplicity and real client-side behaviour in the very same response. No JSON contract, no parallel client app, no duplicated views.

And you may not need a backend at all. JST is just static files and the browser — no API, no server-side code, no build. The same components a server could stream down run perfectly well entirely on the client, so a whole app can be a folder of static files. (This site is exactly that.)

Philosophy

  • No build step. Components live in <script type="jst"> in plain HTML. The browser is the toolchain — nothing to compile, bundle, or transpile.
  • JavaScript is the templating language. $(expr) to interpolate, $ if / $ forEach for control flow. No new DSL — if you know JavaScript, you know JST.
  • Hypertext as the API. Responses are HTML carrying their own UI and next actions — not JSON a client must re-render. Like HTMX, but the response brings its own front-end interactivity.
  • No backend required. A JST app is just static files; the same components run on the client, stream down from a server, or ship as a folder of files. No JSON contract, no parallel client app.
  • It's just Web Components. Every <jst-*> is a real custom element via customElements.define. Data in as properties, actions out as bubbling CustomEvents, DOM patched by morphing — no virtual DOM, no signals, no shadow-DOM lock-in.
  • Props down, events up. Components are dumb renderers; state lives in the page or a parent, recursively.
  • Safe by default. Interpolation is HTML-escaped unless you opt out with raw().

VS Code tooling

Because JST is plain HTML + JavaScript, the editor experience rides existing tools rather than reinventing them. The extension adds three tiers — all tested headlessly.

Syntax highlighting

TextMate injection grammars highlight $(…), $ … and .prop/@event inside <script type="jst">, delegating the embedded JS to the editor's own source.js.

Diagnostics

Runs the real JST compiler over each template, so an editor squiggle is exactly what the runtime would reject. Embedded-JS syntax errors are caught by V8 itself.

Language server

Cross-file go-to-definition (a <my-card> tag → its <script type="jst" name="my-card">), hover with a component's props, completion, and document symbols.
