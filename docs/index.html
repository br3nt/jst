JST docs

JST is for HTML-first apps where the server and browser can share one UI contract: HTML. The server can return pages or fragments; the browser can run those same components with client-side interactivity, no JSON view-model layer, no virtual DOM, and no build step.

The short version

Most web apps turn database state into backend models, then DTOs, then frontend models, then components, then HTML. JST asks whether the backend and browser can both speak the final format directly.

A JST component is just HTML plus JavaScript. The browser registers it as a real custom element. A server can also emit that same component syntax in an ordinary HTML response.

Use full-page RESTful navigation by default. Use fragment swaps only where they make a workflow better: inline validation, load-more feeds, modal bodies, expanding rows, or shipping a component definition the page did not know yet.

Good fit

Static sites, SSR apps, HATEOAS apps, internal tools, progressively enhanced pages, and apps where state naturally belongs in the page or on the server.

Poor fit

Large client-owned SPAs with deep client routing, normalized caches, optimistic updates everywhere, and a need for a mature component ecosystem.

Install

Runtime mode is one module script. It registers every inline JST template and watches trusted inserted fragments.

<script type="module" src="/jst.js"></script>

<script type="jst" name="hello-name" props="name">
  <p>Hello, $(name)!</p>
</script>

<hello-name name="JST"></hello-name>

For local preview, serve the repo root:

python3 -m http.server 8000
# open http://localhost:8000/

Direct file mode

The current runtime is an ES module build. Chrome blocks external module imports from file://, so direct disk-opened pages need either a local static server or a future classic/global build.

Components

Declare props explicitly. Inside the template, props are bare JavaScript locals and element properties.

<script type="jst" name="todo-item" props="item onToggle">
  <li class="$(item.done ? 'done' : '')">
    $(item.text)
    <button @click.stop="$(() => onToggle(item))">Toggle</button>
  </li>
</script>

<todo-item .item="$(todo)" .onToggle="$(handleToggle)"></todo-item>

No separate events="..." declaration is required. Emit DOM events with el.emit(...), listen with @event, and pass direct callbacks as ordinary props when that is clearer.

HATEOAS fragments

A response can define and use a component in the same HTML fragment. When inserted into a trusted auto-register root, the template registers and the element upgrades.

<script type="jst" name="app-notice" props="message level">
  <aside class="notice $(level)">$(message)</aside>
</script>

<app-notice level="info" message="Saved"></app-notice>

Ordinary RESTful routes remain the default: links and forms can navigate to new pages like any SSR app. Fragment swaps are an enhancement for the places where partial updates are worth the extra moving parts.

Server examples

Backend templates can emit JST directly because the wire format is HTML.

Rails ERB

<task-row
  task-id="<%= task.id %>"
  title="<%= h task.title %>"
  done="<%= task.done.to_json %>">
</task-row>

ASP.NET Razor

<script type="jst" name="task-row" props="taskId title done">
  <button @@click="$(() => el.emit('complete', taskId))">Complete</button>
</script>

<task-row
  task-id="@Model.Task.Id"
  title="@Model.Task.Title"
  done="@Json.Serialize(Model.Task.Done)">
</task-row>

Razor uses @@click to render a literal JST @click attribute.

Forms and state

Parent/server-owned state stays explicit:

<input .value="$(title)" @input="$(e => el.emit('title-change', e.target.value))">

Local draft state can use jst-model:

<input jst-model="title">

jst-model reads and writes el.title. It is local host-property sugar, not hidden parent state.

Production

Runtime mode compiles templates in the browser with new Function. Strict CSP should use precompiled mode:

node tools/precompile.mjs components.html --out dist/templates.js --runtime ../jst.js

The generated module registers through the normal runtime, so keyed morphing, form-state preservation, slots, transitions, and modifiers still work.

Security

  • $(expr) escapes HTML by default.
  • url(value) blocks dangerous URL schemes in URL-bearing attributes.
  • raw(value) / unsafeHTML(value) bypass escaping. Only use them for HTML you trust.
  • Fetched JST templates are code. Scope them with autoRegisterRoot and allowlist names in resolveTemplate.

Known gaps

  • No official formatter, tree-sitter grammar, or typed expression checking yet.
  • No marketplace release for the VS Code extension yet.
  • Direct file:// mode needs a classic/global build.
  • SSR hydration plus already-projected light-DOM slots is documented as planned.
  • HTMX-style declarative swap helpers are worth exploring, but not shipped.
